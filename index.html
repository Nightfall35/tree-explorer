01 — Traversal Orders
Visiting Every Node
Four strategies for walking a binary tree. Each has distinct use cases.
Press Play or step through

// Pre-order (Root→L→R)
function preorder(node) {
  if (!node) return
  visit(node)               // ROOT
  preorder(node.left)
  preorder(node.right)
}
// In-order (L→Root→R)
function inorder(node) {
  if (!node) return
  inorder(node.left)
  visit(node)               // ROOT
  inorder(node.right)
}
// Post-order (L→R→Root)
function postorder(node) {
  if (!node) return
  postorder(node.left)
  postorder(node.right)
  visit(node)               // ROOT
}
// Level-order (BFS)
function levelOrder(root) {
  const q = [root]
  while (q.length) {
    const n = q.shift()
    visit(n)
    if (n.left) q.push(n.left)
    if (n.right) q.push(n.right)
  }
}
02 — AVL Trees
Self-Balancing BSTs
Named after Adelson-Velsky and Landis (1962). Every node maintains a balance factor of –1, 0, or +1.
Balance factor = height(left subtree) − height(right subtree). When any node's factor becomes ±2, a rotation is applied to restore balance. This guarantees O(log n) for search, insert, and delete.
bf = 0 (balanced)
bf = ±1 (ok)
bf = ±2 (rotate!)

03 — Rotations
Restoring Balance
Four rotation cases, triggered when balance factor reaches ±2.
LL Rotation
Right rotation — heavy on left-left
RR Rotation
Left rotation — heavy on right-right
LR Rotation
Left then right — heavy on left-right
RL Rotation
Right then left — heavy on right-left
AVL Deletion — Three Cases
CASE 1
Leaf node
Simply remove the node. Update heights up the path. Rebalance if needed.
CASE 2
One child
Replace node with its single child. Update heights and rebalance ancestors.
CASE 3
Two children
Find in-order successor (smallest in right subtree). Copy its value, delete the successor, rebalance.
04 — Operations
Core BST & AVL Operations
Every key operation, its logic, and its time complexity.
Search
O(log n) AVL
Compare key with root. Go left if smaller, right if larger. Repeat until found or null.
Insert
O(log n) AVL
BST insert, then walk back up updating heights. Apply rotation if any node becomes unbalanced.
Delete
O(log n) AVL
BST delete (3 cases). Update heights on ancestors. Rebalance with rotations as needed.
Min / Max
O(log n)
Min: follow left children to the leaf. Max: follow right children. Always O(h) = O(log n) in AVL.
Successor
O(log n)
If right child exists: leftmost node in right subtree. Otherwise: lowest ancestor for which node is in left subtree.
Predecessor
O(log n)
Mirror of successor. If left child exists: rightmost node in left subtree. Otherwise walk up until a right turn.
Height
O(1) cached
Stored at each node as an integer. Updated bottom-up after every insert/delete. height = 1 + max(h_left, h_right).
Balance Factor
O(1)
bf = height(left) − height(right). Must stay in {−1, 0, 1}. Computed from cached heights — O(1).
Range Query
O(log n + k)
Find lower bound, then in-order traverse until upper bound. k = number of results in range.
05 — Complexity
Time & Space Analysis
AVL vs unbalanced BST — why balance matters.

Insert O(log n) O(n) O(log n)

Why O(n) worst case for plain BST?
If you insert values in sorted order (1, 2, 3, 4, 5…), a plain BST degenerates into a linked list — every node has only a right child. Tree height becomes n instead of log n, and all operations take O(n). AVL trees prevent this by enforcing the balance invariant after every operation, guaranteeing height ≤ 1.44 · log₂(n+2) in the worst case.
